## 1. Description Writing Patterns

The description is the only trigger mechanism for a skill. Claude decides whether to use a skill based only on name + description from the `available_skills` list.

### Understanding the Trigger Mechanism

Claude tends not to call skills for simple tasks it can handle easily with its built-in tools. Requests like "read this PDF" may not trigger a skill even with a perfect description. The more complex, multi-step, and specialized a task, the higher the likelihood of skill triggering.

### Writing Principles

1. Describe both **what the skill does** and **specific trigger situations**
2. Specify boundary conditions that distinguish similar but non-triggering cases
3. Be slightly "pushy" — compensates for Claude's conservative tendency to trigger

### Good Examples

```yaml
description: "Read PDF files, extract text/tables, merge, split, rotate, watermark,
  encrypt/decrypt, OCR, and perform all PDF operations. Always use this skill
  when a .pdf file is mentioned or a PDF output is requested. Especially useful
  when conversion/editing/analysis is needed, not just 'read this PDF'."
```

```yaml
description: "All spreadsheet operations including adding columns, formulas, formatting,
  charts, and data cleaning for Excel/CSV/TSV files. Use this skill when the user
  mentions a spreadsheet file — even casually ('the xlsx in the Downloads folder')."
```

### Bad Examples

- `"A skill that processes data"` — too vague, unclear which files/operations
- `"PDF-related tasks"` — no specific operation list, no trigger situation

## 2. Body Writing Style

### Why-First Principle

When LLMs understand the reason, they judge edge cases correctly. Conveying context is more effective than imposing rigid rules.

**Bad:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good:**
```markdown
Use pdfplumber for table extraction. PyPDF2 is optimized for text extraction
and cannot preserve the row/column structure of tables. pdfplumber
recognizes cell boundaries and returns structured data.
```

### Generalization Principle

When a problem is found in feedback or test results, generalize at the **principle level** rather than applying a narrow fix for the specific example.

**Overfitting fix:**
```markdown
If a column named "Q4 Revenue" exists, convert that column to a number.
```

**Generalized fix:**
```markdown
If a column name contains keywords implying numeric values such as "revenue",
"amount", or "quantity", convert that column to a numeric type.
On conversion failure, retain the original value.
```

### Imperative Tone

Use "do X", "return Y" forms rather than "X is done" or "can do Y". Skills are directives.

### Context Conservation

The context window is a shared resource. Ask whether every sentence justifies its token cost:
- "Does Claude already know this?" → delete
- "Would Claude make a mistake without this explanation?" → keep
- "Would a single concrete example be more effective than a long explanation?" → replace with example

## 3. Output Format Definition Patterns

Use this in skills where the format of deliverables matters:

```markdown
## Report Structure
Follow this template exactly:

# [Title]
## Summary
## Key Findings
## Recommendations
```

Keep format definitions concise; including actual examples makes them more effective.

## 4. Example Writing Patterns

Examples are more effective than long explanations:

```markdown
## Commit Message Format

**Example 1:**
Input: Add JWT token-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: Fix bug where password show button on login page doesn't work
Output: fix(login): fix password visibility toggle button
```

## 5. Progressive Disclosure Patterns

### Pattern 1: Domain-based separation

```
bigquery-skill/
├── SKILL.md (overview + domain selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)
```

If the user asks about revenue, load only finance.md.

### Pattern 2: Conditional detail

```markdown
# DOCX Processing

## Document Generation
Create new documents with docx-js. → See [DOCX-JS.md](references/docx-js.md).

## Document Editing
For simple edits, modify XML directly.
**If track changes are needed**: See [REDLINING.md](references/redlining.md)
```

### Pattern 3: Large reference file structure

Reference files over 300 lines should include a table of contents at the top:

```markdown
# API Reference

## Table of Contents
1. [Authentication](#authentication)
2. [Endpoint List](#endpoint-list)
3. [Error Codes](#error-codes)
4. [Rate Limits](#rate-limits)

---

## Authentication
...
```

## 6. Script Bundling Decision Criteria

Observe agent transcripts during test runs. Bundle when these patterns appear:

| Signal | Action |
|--------|--------|
| Same helper script created in 3 out of 3 tests | Bundle in `scripts/` |
| Same pip install/npm install run every time | Specify dependency installation step in skill |
| Same multi-step approach repeated every time | Document as standard procedure in skill body |
| Same error followed by same workaround every time | Document known issues and solutions in skill |

Bundled scripts must pass execution tests.

## 7. Data Schema Standards

Use standard schemas for consistent data exchange between skills. Can be used for testing/evaluation of skills generated by the harness.

### eval_metadata.json

Metadata for each test case:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "User's task prompt",
  "assertions": [
    "Output contains X",
    "File was created in Y format"
  ]
}
```

### grading.json

Assertion-based grading results:

```json
{
  "expectations": [
    {
      "text": "Output contains 'Seoul'",
      "passed": true,
      "evidence": "Confirmed 'Seoul region data extraction' in step 3"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field name note:** Use `text`, `passed`, `evidence` exactly — no variations like `name`/`met`/`details`.

### timing.json

Execution time/token measurements:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Save `total_tokens` and `duration_ms` immediately from the subagent completion notification. This data is only accessible at notification time and cannot be recovered later.

## 8. What Not to Include in Skills

- Supplementary documents such as README.md, CHANGELOG.md, INSTALLATION_GUIDE.md
- Meta information from the skill creation process (test results, iteration history)
- User-facing documentation (skills are directives for AI agents)
- General knowledge Claude already knows
